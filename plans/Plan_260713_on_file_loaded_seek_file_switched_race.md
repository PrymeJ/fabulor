Fix design: the general _on_file_loaded seek/file_switched race
Context
This is fix design for a general, pre-existing, already-instrumented-and-confirmed race condition in _on_file_loaded's "issue a seek, then unconditionally emit file_switched" pattern — it is not a continuation of the VT restore-on-load fix's design (_vt_restore_pending/ defer_vt_restore, implemented earlier this session). That fix is done, tested, and currently sitting on the same branch, uncommitted, blocked on this finding (see "Dependency direction" below). This plan addresses the broader mechanism that fix's own testing surfaced.

Full investigation trail, with live-instrumented timestamped evidence (not just static code tracing), is written up in NOTES.md under "_on_file_loaded's general 'issue a seek, then unconditionally emit file_switched' race" (2026-07-13) — read it before implementing. This plan does not re-derive that evidence; it designs the fix on top of it.

What "solved" means for this fix — stated explicitly, per direct instruction
This race was shown tonight to be broader than the bug that led to its discovery. A fix that only makes the VT-restore-on-load case stop failing, without re-verifying the general manual-seek exposure, would be solving the symptom that got discovered — not the mechanism that was found. This plan claims to fix both, and each has its own concrete, forced-verification bar:

The VT restore-on-load case (the bug that led here): a VT book with a saved position opens and stays at the correct position — not "restores, then resets." Verification: the SAME forced-condition discipline as _vt_restore_pending's own harness (tools/ vt_restore_race_harness.py, still present, temporary) — 100% success under a harness that deterministically forces the race window open, not just repeated real launches.
The general manual-seek case (what tonight's [FS-RACE] instrumentation actually found): every manual VT cross-file-seek trigger — wheel-scroll, arrow-key nav, seek/skip buttons, progress-slider click, chapter-list click — settles correctly and does not orphan _seek_target, under the SAME forced-condition bar, not just "tried it a bunch of times and it looked fine." Tonight's six confirmed live misses (wheel, arrow, seek-button, slider-click, chapter-list-click — exact targets/timestamps in NOTES.md) are the regression set this fix must close, with each one specifically re-tested, not just "the mechanism should now prevent this class of miss."
A fix that only satisfies (1) is incomplete. Both must be independently forced and independently verified before this is considered done.

Root cause (restated from NOTES.md, not re-derived)
_on_file_loaded's VT branch (player.py, currently 606-618) can issue a seek — either via the _pending_local_pos branch (555-601, used by seek_async's VT cross-file branch — manual cross-file seeks) or via consuming _vt_restore_pending (612-615, the VT-restore-on-load fix) — and then unconditionally emits self.file_switched.emit() (616). file_switched connects via Qt.ConnectionType.QueuedConnection to _on_vt_file_switched (app.py:1430-1432), which unconditionally clears self.player.is_seeking = False. Because the connection is queued, the clear runs on the next Qt event-loop iteration — with no guarantee the seek has settled (settling requires an mpv time-pos sample landing within 1.0 of _seek_target, in _on_time_pos_change's settle branch, player.py ~171-183). If the clear wins the race, _seek_target is orphaned (the settle branch requires is_seeking=True to ever fire) and the _logical_pos maintenance block's gate opens prematurely, corrupting position from stale/mid-flight raw mpv samples.

Confirmed via live [FS-RACE] timestamped instrumentation (not just structural reasoning): this race is real and reproducible on the manual cross-file-seek path across every tested input method, at a rate consistent with genuine timing-sensitivity — not deterministic per input, not purely theoretical.

What's closed — explicitly forbidden starting points, not just references
Both are named here with their actual failure reasons stated, not just cited, because this is the third design pass tonight in the same neighborhood and the risk of re-proposing a closed idea under different phrasing is real (it already nearly happened once this session).

Defer file_switched's emission (delay it until the seek settles, or until otherwise safe). CLOSED. Tried 2026-06-06, reverted. Failure reason: broke Undo — the VT slider got stuck after an undo action. This is a documented, load-bearing prior failure in CLAUDE.md's VT+Undo fragile-zone rule. Do not re-propose any variant of "change when file_switched fires."
Narrow _on_vt_file_switched's clear unconditionally (e.g., a blanket guard added to the consumer with no distinction for why) — tried twice this session's drift-fix branch, reverted both times. Failure reason, now precisely understood (not just "it broke"): both prior attempts were tested exclusively against seeks that were structurally incapable of ever landing at all — the VT-restore-on-load book_ready-before-play() bug (Layer 1, now separately fixed this session via _vt_restore_pending/defer_vt_restore), where mpv's raw time_pos was confirmed via instrumentation to never move off ~0. Against a seek that can never land, ANY guard that keeps is_seeking=True alive produces a permanent freeze — that is not evidence against the guard's logic, it is the inevitable consequence of testing it against an unrelated, unfixed bug. This distinction is real and evidenced (git history + this session's own TODO.md/ NOTES.md entries), not a rationalization — see "Why the guard is different this time" below.
A structurally-identical third variant also considered and rejected tonight: conditionally skipping file_switched.emit() itself (not delaying it, not touching the consumer) when a fresh seek was just issued in the same call. Rejected on the behavior-equivalence test: for the "seek never settles" case, this produces the exact same permanent-freeze outcome as approach 2 — the consumer never runs either way, so "emission skipped" and "emission fired but declined" are behaviorally indistinguishable. Not listed as a third closed approach in CLAUDE.md (never implemented/tested), but ruled out here on the same grounds and should not be re-proposed as if it were meaningfully different from approach 2.
A synchronous mpv property read (self.instance.time_pos) issued immediately after the seek command, to short-circuit needing to wait for the async settle. Investigated and ruled out with evidence (python-mpv source): command_async/synchronous command both only guarantee the seek was dispatched/accepted, never that it has landed — mpv's seek pipeline is asynchronous internally regardless of client-API call style. A sync read right after issuing the seek would sometimes get lucky (near-file-start seeks) and sometimes read stale/mid-flight data — reintroducing the same race under a different guise, not solving it.
The fix — gate _on_vt_file_switched's clear, PLUS a required companion fix
Part 1: gate _on_vt_file_switched's clear on _seek_target is None
_on_vt_file_switched (app.py:1430-1432) changes from an unconditional self.player.is_seeking = False to a guarded clear: only clear when self.player._seek_target is None.

Why this is the correct fix, not a repeat — the data-model finding (investigated tonight, not assumed): every reader of is_seeking in the codebase (_sync_progress_sliders, _sync_chapter_ui's seek guard, _update_chapter_label_from_index, _update_ui_sync's paused-time logic) gates on "has the seek settled to its target" semantics — none of them need "file switch is done" as a distinct trigger. _on_vt_file_switched's clear is git-confirmed to have been added (1c8d1b6, 2026-05-15) on top of an already-working settle mechanism (_seek_target/the settle branch, introduced the prior commit, 7f891f1) — it is a second, competing writer to state the settle branch already owns correctly whenever a seek was actually issued. Its only genuinely necessary case is the no-seek-issued path (natural EOF-advance, or a restore with no saved progress), where _seek_target stays None and the settle branch will never fire on its own — exactly the case the is None guard preserves.

Why the two prior freezes are not evidence against this guard, checked (not assumed): both were tested exclusively against Layer-1-broken seeks — confirmed via this session's own TODO.md entry ("mpv never moves off ~0 ... the seek never settles to clear it, since mpv is at 0") and via git history showing no other guard attempt ever existed. Neither attempt was ever tested against a seek proven capable of landing (tonight's [FS-RACE] evidence shows the seek DOES land in the general race's failure cases — it's a timing loss, not a structural non-landing). This guard has never actually been tested under the conditions it will now run in.

Part 2 (REQUIRED, not optional): close the one real remaining freeze gap first
Investigated tonight, specifically to answer "is there still a live way a seek can genuinely never settle, independent of the two now-understood/fixed causes" — yes, one confirmed, currently-open gap exists on main today, unrelated to either bug already addressed this session:

_on_end_file's ERROR path never clears seek state. _on_end_file (player.py:620-629):

def _on_end_file(self, event):
    data = event.data
    reason_int = data.reason if data else -1
    if reason_int == 4:  # ERROR
        error_str = event.as_dict().get('file_error', b'').decode('utf-8', errors='replace')
        detail = error_str if error_str else 'unknown error'
        self.load_failed.emit(detail)
    if reason_int == 0:
        self._advance_or_finish()
If a VT cross-file seek's instance.play(target_file['file_path']) (seek_async's cross-file branch, 793) or _advance_or_finish's instance.play(next_file['file_path']) (289) targets a file that fails to load (missing from disk, corrupt, permissions error — anything mpv reports as end-file reason=ERROR), file-loaded never fires, so neither _on_file_loaded nor _on_time_pos_change's settle branch ever runs. _on_load_failed (app.py:1619-1624) only shows a banner and conditionally calls _mark_book_missing — it does not touch player.is_seeking/ _seek_target. If a seek was in flight when this happens, is_seeking stays True forever, with no settle and (once Part 1 lands) no _on_vt_file_switched clear either, since the guard requires exactly the condition (_seek_target is not None) that this scenario leaves permanently true. This must be fixed as part of the same change, not after — landing Part 1 without Part 2 would introduce a new, real freeze path that wasn't reachable before (today, the unconditional _on_vt_file_switched clear at least recovers from this case, badly, by clobbering state; after Part 1 alone, it wouldn't recover at all).

Fix: in _on_end_file's reason_int == 4 branch, also clear seek state — self._is_seeking = False, self._seek_target = None, and reset _logical_pos/_last_raw_global/_just_settled the same way the existing near-EOF seek-abandon path does (_on_file_loaded ~568-577, the _pending_local_pos "abandoned" branch — mirror that exact reset shape, don't invent a new one). This is a narrow, targeted addition to a branch that currently does nothing but emit a signal — it does not touch the reason_int == 0 (natural EOF) branch, _advance_or_finish, or any other part of _on_end_file.

Other items from tonight's enumeration, classified but explicitly NOT required for this fix (real but separately scoped, or not applicable):

command_async('seek', ...) being silently rejected by mpv with no error signal at all — a theoretical, unconfirmed-live gap (no reproduction exists). Not fixed here; if it's ever observed live, it needs its own investigation, not a preemptive guess bolted onto this change.
Overlapping seeks (a second seek_async call while a first is still in flight) — confirmed safe; worst case is a spurious early settle (harmless), not a freeze. No change needed.
The deferred-VT-restore-landing-within-2s-of-EOF concern — not an actual gap; it re-enters seek_async, inheriting that function's own existing 2s guards. No change needed.
Dependency direction relative to _vt_restore_pending — decided, not left implicit
This fix (Parts 1 and 2) must land before _vt_restore_pending's own verification can be considered complete — not the other way round, and not as two independently-landable changes. _vt_restore_pending's reliability was shown tonight to depend on this exact mechanism (it's what caused the ~4 real "restores then resets" failures out of ~50-90 launches). Landing _vt_restore_pending alone, without this fix, ships a fix that's known to still fail intermittently for a reason now fully understood and just not yet corrected. Sequencing:

Implement Parts 1 and 2 of this plan.
Re-run _vt_restore_pending's own full verification bar (the forced-delay harness, the 10-15 repeated real launches, the VT+Undo standing-rule checklist) — it should now pass without the intermittent failures seen before, because the mechanism causing them is fixed.
Separately and additionally, re-run the general-race verification (six-input-method forced reproduction, see below) to confirm the manual-seek exposure is also closed.
Only after both (2) and (3) pass does the combined change (this fix + _vt_restore_pending) get committed/considered complete. They land together, verified together, not staged independently.
What must NOT change
file_switched's emission timing/site — _on_file_loaded keeps emitting it exactly where and when it does today (line ~616 area). This fix changes what the CONSUMER does with it once received, gated on state already available at consumption time — not when or whether it fires.
The settle branch and _logical_pos maintenance block in _on_time_pos_change — untouched. This fix makes the guard consistent with what they already assume (only clear is_seeking when nothing is genuinely pending) rather than changing their own logic.
_vt_restore_pending/defer_vt_restore — untouched by this plan; this fix is what makes that existing code reliable, not something that requires changing it further.
The non-VT restore/seek path — _on_vt_file_switched is VT-only (file_switched only ever fires from the VT branch); this fix has no surface area on non-VT books.
_advance_or_finish's natural-EOF-advance branch (reason_int == 0 in _on_end_file) — only the reason_int == 4 (ERROR) branch gains new logic; natural advance is already confirmed unaffected by any of tonight's findings.
Verification — the same forced-condition bar, applied to BOTH claims from "What solved means"
A single clean live pass does not prove this fix works — the race is timing-dependent by nature, proven tonight to sometimes settle cleanly and sometimes miss on the identical input, identical book, identical session. Verification must force both failure conditions this fix addresses, not just retry until it looks fine:

Reuse tools/vt_restore_race_harness.py (still present, temporary, from the earlier _vt_restore_pending work) to re-confirm the VT-restore-on-load case under forced conditions, now with Parts 1+2 applied — 100% success required, matching the harness's own established bar (sub-steps 1a/1b/1c: inert-when-disabled, forces the pre-fix code to fail, confirms the fixed code succeeds).
A NEW forced-condition test for the general manual-seek case, since no harness currently exists for this — design one (deterministic, not sleep-based, following the same _ControllableMpv-style mock pattern the existing harness uses) that can force the exact race window tonight's [FS-RACE] captures showed: a seek genuinely landing, with the _on_vt_file_switched clear arriving at a controllable, adversarial moment relative to the settle. Prove: with the OLD (unguarded) code, the clear-before-settle ordering corrupts state (matching tonight's 6 confirmed misses); with the NEW (guarded) code, the same ordering does NOT corrupt state, regardless of which order the two events arrive in.
A dedicated forced test for Part 2 (the _on_end_file ERROR-path fix): simulate a VT cross-file seek issued against a file that then fails to load (mock instance.play to trigger an ERROR end-file event) — confirm is_seeking/_seek_target are cleared afterward, not left stuck, with and without Part 1's guard active (this is the specific case that would newly freeze if Part 2 were skipped).
Repeated real launches AND repeated real manual cross-file seeks, no artificial delay — the same six input methods tonight's [FS-RACE] pass covered (wheel, arrow, seek-button, progress-slider click, chapter-list click, and natural EOF-advance as a control) — enough repetitions to have real confidence, not a fixed arbitrary count; if any single miss is observed, stop and report per the escalation conditions below, do not treat it as noise.
The full VT+Undo standing-rule checklist (CLAUDE.md), run BEFORE declaring either presenting symptom fixed, per the standing rule's required order: VT normal playback across a file boundary, VT cross-file seeking, and Undo — all live-verified for regressions, since this is now the FIFTH documented touch on this exact fragile zone this session (four in the drift-fix + VT-restore history, this being the fifth).
pytest tests/ -q green, plus new unit tests for _on_vt_file_switched's guarded clear (pure state-machine style, no mpv/QApplication, matching tests/test_vt_seek.py's existing pattern) covering: clear fires when _seek_target is None; clear does NOT fire when _seek_target is not None; and the _on_end_file ERROR-path reset (Part 2) restores is_seeking=False/_seek_target=None correctly.
Escalation conditions — stop and report, do not push through
If the new forced-condition test for the general manual-seek case cannot be made to reliably reproduce the OLD code's clobber under a controlled, adversarial ordering — stop and report; that means confidence in this fix cannot be established the way this plan intends.
If VT+Undo live verification turns up ANY regression, however small or apparently unrelated — stop and report immediately, do not patch inline. This is the standing rule for this exact fragile zone, restated because this is its fifth touch tonight.
If implementing Part 2 reveals that _on_end_file's ERROR path needs to touch anything beyond is_seeking/_seek_target/_logical_pos/_last_raw_global/_just_settled (e.g., if the existing near-EOF-abandon reset pattern this mirrors turns out to be an incomplete model for the ERROR case specifically) — stop and report the discrepancy rather than inventing a new reset shape ad hoc.
If re-running _vt_restore_pending's own verification bar (step 2 of the dependency sequencing above) still shows intermittent failures after this fix lands — that means the mechanism understanding is incomplete, not that the failure rate was "close enough." Stop and report; do not conclude the fix is "mostly working."
