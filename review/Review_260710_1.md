## 2. Scroll-position content-anchoring fix

### 2a. Capture-before / scroll-after inside the existing deferral — PASS
- `top_row = self._first_visible_row()` is captured **before** `self._apply_view_mode(mode)`
  ([library.py:1160-1162](src/fabulor/ui/library.py#L1160-L1162)).
- `scrollTo(idx, PositionAtTop)` runs **inside the existing** `_after_reset` closure that is already
  fired via `QTimer.singleShot(0, _after_reset)` ([library.py:1177-1183](src/fabulor/ui/library.py#L1177-L1183))
  — not a new deferred-wait mechanism. The 5-attempt `visualRect` retry that already existed is reused.
- `_first_visible_row()` ([library.py:1231-1255](src/fabulor/ui/library.py#L1231-L1255)) is the shared
  helper; `_load_visible_covers` calls the same one ([library.py:1257](src/fabulor/ui/library.py#L1257)),
  so capture and cover-loading agree on "what's on top."

### 2b. No raw scrollbar pixel / fraction carry remains — PASS
The captured value is a plain model index into `_filtered`. Nothing in `_on_view_mode_changed`
reads or writes `verticalScrollBar().value()` or any range fraction. The geometry-anchored carry
(Qt's default) that the commit replaced is gone from this path.

### 2c. Bounded, never out-of-range — PASS
`scrollTo` is guarded by `top_row is not None and 0 <= top_row < self._book_model.rowCount()`
([library.py:1177](src/fabulor/ui/library.py#L1177)) and `idx.isValid()`. The index is captured from
the same model the switch does not reorder/refilter, so it is always in range; the explicit bound
makes an out-of-range `scrollTo` impossible. The accepted ±1–2 row grid↔grid drift is inherent to
differing column counts (same book index → different row/column packing), not a symptom of a larger
problem — a valid in-range `PositionAtTop` scroll cannot land arbitrarily far off.

## Cross-change interaction (ESCALATE clause) — no interference

Two changes in this window both now touch panel-animation state: the hover debounce (fix 3) and the
preloader's `_preload_paused` gate. They do **not** step on each other:
- The hover debounce is a standalone `QTimer` in `ThemeManager`; it does not mutate panel-animation
  state and is never read by the preloader.
- `_preload_paused` only *reads* `pm.is_any_panel_animating()`, `_fade_anim.state()`, and
  `_fade_in_flight` — never writes them.
- The only shared surface is `_fade_anim`: the hover fix *starts* it after restyle; the preloader
  *reads* its state to pause. That is the intended relationship (pause background LANCZOS while a
  hover-preview fade is animating). No shared mutable state, no ordering hazard.

## Observations (not defects; not introduced by this window; no action taken)

**A. Raw-cover cap asymmetry between the two raw-cover slots.**
`_on_cover_loaded` caps the raw cover to ≤320×480 before caching
([library.py:1306-1312](src/fabulor/ui/library.py#L1306-L1312)); `_on_preload_cover_loaded` does
**not** ([library.py:1654-1660](src/fabulor/ui/library.py#L1654-L1660)) — the worker loads the raw
image uncapped. So the same book's `_cover_cache` entry can differ in resolution depending on which
slot warmed it. This is a **pre-existing** asymmetry (the preloader-raw slot never capped), not
introduced by these commits, and it does not break the sized-cache keying (the sized entry is keyed
and returned before `_get_sized_cover` ever inspects the raw cover). Its only effect is that a
preloaded sized pixmap may be LANCZOS-scaled from a marginally higher-res source than the on-demand
path would use — a quality nuance, not a correctness issue. Flagging for awareness only.

**B. Multi-screen DPR source difference.**
`_get_sized_cover` derives DPR from `cover.devicePixelRatio()` while the preloader derives it from
`self.screen().devicePixelRatio()`. These agree on a single screen (the raw pixmap is stamped with
the screen DPR at load). On a genuine multi-screen setup with differing DPRs they could diverge and
a preloaded sized entry could miss — but this is the already-tracked "Screen drag 4K→1080p: cover
scaling doesn't update without scroll" debt in CLAUDE.md, not a regression from this window.
