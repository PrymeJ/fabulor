## 1. `_on_time_pos_change` seek-clear logic ÔÇö (a) correct

[player.py:112-116](src/fabulor/player.py#L112-L116):
```python
if self._is_seeking and value is not None and self._seek_target is not None:
    global_value = value + (self._file_offset or 0)
    if abs(global_value - self._seek_target) < 1.0:
        self._is_seeking = False
        self._seek_target = None
```
`_is_seeking` is cleared **only** inside the `self._seek_target is not None` branch AND only when `abs(global_value - self._seek_target) < 1.0`. There is no path in this method that clears `_is_seeking` when `_seek_target is None`. The asymmetric-clear bug (clearing on the first `time_pos=0` of a fresh load) is correctly avoided. `load_book` also resets `_seek_target = None` ([player.py:360](src/fabulor/player.py#L360)). Ôťů

The other `_is_seeking = False` site is `_on_file_loaded`'s mp3-reload early-return ([player.py:417](src/fabulor/player.py#L417)), which is correct (the reload's `_seek_target` was set by `_mp3_stop_and_load`). Ôťů

## 2. Seek / loadfile 2.0s-of-EOF guards ÔÇö (a)* correct, with a note

Four seek/loadfile sites:

| Site | file:line | Guard | Verdict |
|------|-----------|-------|---------|
| Non-VT `seek_async` | [player.py:568](src/fabulor/player.py#L568) | `if dur and dur - pos < 2.0: return` at [player.py:560](src/fabulor/player.py#L560) | Ôťů direct guard |
| VT same-file `seek_async` | [player.py:548](src/fabulor/player.py#L548) | `if target_file['duration'] - local_pos < 2.0: return` at [player.py:546](src/fabulor/player.py#L546) | Ôťů direct guard |
| `_mp3_stop_and_load` loadfile | [player.py:523](src/fabulor/player.py#L523) | non-VT caller requires `> _MP3_SEEK_THRESHOLD` (60s) seek; VT caller requires `2.0 < local_pos < target_file['duration'] - 5.0` ([player.py:542](src/fabulor/player.py#L542)) | Ôťů 5s buffer on the path that can land near EOF |
| Pending-seek after VT file switch | [player.py:426](src/fabulor/player.py#L426) | **no direct guard** ÔÇö see note | ÔÜá indirect |

**Note (medium):** The pending-seek at [player.py:426](src/fabulor/player.py#L426) (`_on_file_loaded` consuming `_pending_local_pos`) has no inline `duration - pos < 2.0` check. It is reached via the VT cross-file branch of `seek_async` ([player.py:549-557](src/fabulor/player.py#L549-L557)), where `_pending_local_pos = pos - target_file['cumulative_start']` is a *local* offset into a *newly-switched* file. Because a cross-file seek lands the user near the *start* of the target file (the resolved `target_idx` is the file whose `cumulative_start <= pos`), `local_pos` is normally far from that file's EOF, so the hang condition is not realistically hit. It is nonetheless the one seek path without its own guard. This matches the documented guard inventory in CLAUDE.md ("Guards currently live in `seek_async`"), so it is **not a rule violation** ÔÇö but if a future change lets a cross-file seek target land within 2s of the *target file's* duration, this path would hang silently. Recommend a defensive `duration - pending < 2.0` check at [player.py:425](src/fabulor/player.py#L425) when convenient. Severity medium only because it is currently unreachable in practice.

The VT cross-file `instance.play()` at [player.py:557](src/fabulor/player.py#L557) is a file load, not a seek into the current stream, so the EOF-hang rule does not apply to it.
