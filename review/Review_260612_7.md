## 1. `open()` unconditional reset — (a) correct

[session_recorder.py:59-74](src/fabulor/session_recorder.py#L59-L74). `open()` sets, with no conditional gating, at the top of the method:
- `self._post_seek_pending_position = None` ([session_recorder.py:73](src/fabulor/session_recorder.py#L73))
- `self._seek_credit_timer.stop()` ([session_recorder.py:74](src/fabulor/session_recorder.py#L74))

Both run on **every** `open()` call regardless of entry path. The inline comment ([session_recorder.py:69-72](src/fabulor/session_recorder.py#L69-L72)) documents exactly the hazard this guards: a fresh `open()` while a prior forward-seek's 15s credit timer is still live would otherwise inherit a stale `_post_seek_pending_position`, which blocks every `update_furthest_position` tick. `resume()` ([session_recorder.py:82-85](src/fabulor/session_recorder.py#L82-L85)) deliberately does **not** reset these (correct — resume continues the *same* session, so the pending seek window should persist). The three entry paths:
- **Fresh open** (first play, no active session): `app.py` calls `open()` → full reset.
- **Resume after short pause**: calls `resume()` → no reset (intended; same session).
- **Re-open after sleep-timer pause that timed out**: once the 3-min `_pause_timer` fires `close()` ([session_recorder.py:37](src/fabulor/session_recorder.py#L37)), `is_active` is False, so the next play calls `open()` → full reset.

`close()` also resets `_post_seek_pending_position = None` ([session_recorder.py:159](src/fabulor/session_recorder.py#L159)), so the state can never leak across a close→open boundary either. ✅

## 2. Sleep-timer ↔ recorder: no double-open/double-pause — (a) correct

Three sleep-timer signals are wired in `app.py` ([app.py:565-567](src/fabulor/app.py#L565-L567)): `timer_started → _on_sleep_timer_started`, `timer_stopped → _on_sleep_timer_stopped`, `timer_expired → _on_sleep_timer_expired`.

**Start direction** — `_on_sleep_timer_started` ([app.py:657-666](src/fabulor/app.py#L657-L666)):
```python
if self.current_file:
    if not self.session_recorder.is_active:
        self.session_recorder.open()
    else:
        self.session_recorder.resume()
```
The `is_active` check ([app.py:663](src/fabulor/app.py#L663)) is the double-open guard: it `open()`s only when no session exists, else `resume()`s. So starting a sleep timer while a session is already active cannot double-open. ✅

**Expiry direction** — `_on_sleep_timer_expired` ([app.py:675-681](src/fabulor/app.py#L675-L681)):
```python
if self.session_recorder.is_active:
    self.session_recorder.pause()
```
Guarded by `is_active` ([app.py:680](src/fabulor/app.py#L680)) — `pause()` only fires on a live session. **Double-pause safety:** even if `pause()` were somehow called twice, it is idempotent against double-accumulation — `pause()` ([session_recorder.py:87-94](src/fabulor/session_recorder.py#L87-L94)) only accumulates `if self._session_segment_start is not None` ([session_recorder.py:89](src/fabulor/session_recorder.py#L89)) and sets it to `None` ([session_recorder.py:92](src/fabulor/session_recorder.py#L92)); a second `pause()` finds it already None and accumulates nothing (it would only restart the 3-min timer, harmless). So "sleep timer fires while the book is already paused for another reason" cannot double-count listened time: the prior pause already nulled `_session_segment_start`. ✅

`_on_sleep_timer_stopped` ([app.py:668-673](src/fabulor/app.py#L668-L673)) is UI/fade only — it does not touch the recorder, correctly (a user cancelling the timer should not open/close/pause a session). ✅

## 3. `started_at` not conflated with streak first-session source — (a) correct

`set_started_at`/`get_book_started_at` ([db.py:584-602](src/fabulor/db.py#L584-L602)) read/write `books.started_at`, keyed by `book_id`. They are called from exactly one place — `SessionRecorder.close()`/`_recover_checkpoint` ([session_recorder.py:143-144](src/fabulor/session_recorder.py#L143-L144), [session_recorder.py:244-245](src/fabulor/session_recorder.py#L244-L245)) — to stamp "when the user first started this book," and read for the book-detail "started" metadata. **No streak query reads `started_at`.**

The streak computation derives from `session_start`, not `started_at`:
- `get_streaks` ([db.py:937-975](src/fabulor/db.py#L937-L975)) builds its day set from `self.get_active_periods('day', day_start_hour)` ([db.py:945](src/fabulor/db.py#L945)), which is `SELECT DISTINCT strftime(..., session_start, ...) FROM listening_sessions` ([db.py:619-625](src/fabulor/db.py#L619-L625)).
- The streak grid cache (`build_streak_grid_cache` [db.py:992-1034](src/fabulor/db.py#L992-L1034), `_update_streak_grid_cache_for_date` [db.py:1056-1078](src/fabulor/db.py#L1056-L1078)) keys off `session_start`/`session_end` adjusted dates.

So "first session of the day" for streak purposes comes from `listening_sessions.session_start`, never `books.started_at`. No conflation in either direction. ✅

## 4. `delete_session` deletes by unique `id`; no stale cross-table reference — (a) correct

`delete_session(session_id, day_start_hour)` ([db.py:896-912](src/fabulor/db.py#L896-L912)):
```sql
DELETE FROM listening_sessions WHERE id = ?
```
([db.py:909](src/fabulor/db.py#L909)) — deletes by the **`id` INTEGER PRIMARY KEY AUTOINCREMENT** ([db.py:75](src/fabulor/db.py#L75)), a unique key, **not** a `book_id`+timestamp compound. The `session_id` is sourced from the row dict produced by `get_book_sessions` (`SELECT id, ...` [db.py:881-893](src/fabulor/db.py#L881-L893)) and carried through `_HistoryRow` → `_on_history_delete_confirmed` ([book_detail_panel.py:1080-1085](src/fabulor/ui/book_detail_panel.py#L1080-L1085)).

**No other table holds a session-id reference.** `listening_sessions.id` is not a foreign key anywhere: `book_events` references `books(id)` not `listening_sessions(id)` ([db.py:103-108](src/fabulor/db.py#L103-L108)); `streak_grid_cache` is keyed by `date` ([db.py:144-147](src/fabulor/db.py#L144-L147)), holding no session id; aggregates (`get_book_stats`, `get_overall_stats`) compute on the fly from `listening_sessions` rows and hold no persisted id. So deleting a session row leaves no dangling reference — the only derived artifact (the streak cache cell) is recomputed in the same transaction (item 5). ✅

## 5. Streak grid source + `delete_session` invalidation; no cartesian recompute — (a) correct

**Source:** the `StreakGrid` widget reads the **precomputed `streak_grid_cache` table**, not `listening_sessions` directly. `_refresh_time` ([stats_panel.py:2613-2618](src/fabulor/ui/stats_panel.py#L2613-L2618)):
```python
cache = self.db.get_streak_grid_cache()        # SELECT date, listened FROM streak_grid_cache
streak = self.db.get_streaks(...)
finished = self.db.get_streak_grid_finished_dates()
self._streak_grid.set_data(cache, streak, finished, datetime.now().date())
```
`get_streak_grid_cache` ([db.py:1036-1042](src/fabulor/db.py#L1036-L1042)) reads the cache table.

**Invalidation on delete:** because there IS a precomputed table, the rule is that `delete_session` must recompute the affected day. It does ([db.py:910-912](src/fabulor/db.py#L910-L912)):
```python
if row is not None:
    for d in {row[0], row[1]}:  # start-day + end-day, dedup
        self._update_streak_grid_cache_for_date(conn, d, day_start_hour)
```
Both the deleted session's adjusted start-day and end-day cells are re-evaluated, **inside the same `with self._get_conn()` transaction** as the DELETE ([db.py:902-912](src/fabulor/db.py#L902-L912)) — the docstring's "single transaction so cache never disagrees with rows" ([db.py:899-900](src/fabulor/db.py#L899-L900)) holds. A midnight-spanning session correctly clears both its days.

**No cartesian-product hazard:** `_update_streak_grid_cache_for_date` ([db.py:1069-1074](src/fabulor/db.py#L1069-L1074)) is a plain `SELECT COUNT(*) FROM listening_sessions WHERE strftime(...session_start...) = ? OR strftime(...session_end...) = ?` — it touches **only** `listening_sessions`, never joins `book_events`. So the CLAUDE.md "DO NOT join `book_events` into a query that aggregates `listening_sessions`" rule is not at risk here. The recompute is `1 if count > 0 else 0` ([db.py:1075-1078](src/fabulor/db.py#L1075-L1078)). ✅

## 6. One `day_start_hour` config source for streak + stats — (a) correct

Both surfaces read `self.config.get_day_start_hour()` — there is **no** hardcoded midnight on either side:

- **Stats panel period queries:** `get_active_periods(..., self.config.get_day_start_hour())` and `get_daily_book_breakdown(date_str, self.config.get_day_start_hour())` ([stats_panel.py:2140](src/fabulor/ui/stats_panel.py#L2140), [2165](src/fabulor/ui/stats_panel.py#L2165), [2295](src/fabulor/ui/stats_panel.py#L2295), [2452](src/fabulor/ui/stats_panel.py#L2452), [2542](src/fabulor/ui/stats_panel.py#L2542)).
- The spin control writes the one config value and triggers a rebuild: `day_start_spin.valueChanged.connect(self.config.set_day_start_hour)` + `_on_day_start_hour_changed` ([stats_panel.py:1861-1862](src/fabulor/ui/stats_panel.py#L1861-L1862)), the latter rebuilding the streak cache against the new hour ([stats_panel.py:2602-2611](src/fabulor/ui/stats_panel.py#L2602-L2611)).

The DB layer itself never reads config — `day_start_hour` is always passed in as a parameter (consistent contract across `db.py`, documented at [db.py:987-988](src/fabulor/db.py#L987-L988)). Streak and stats are therefore guaranteed consistent: they pull from the same `config.get_day_start_hour()` accessor. No inconsistency to flag. ✅

## 7. Finished-book ↔ session-delete — (a) correct

**Grid updates when the day's only session is deleted:** covered by item 5 — `delete_session` recomputes the affected day's cache cell via `COUNT(*)`; if that was the last session for the day, the count is 0 and the cell flips to `listened = 0` ([db.py:1075-1078](src/fabulor/db.py#L1075-L1078)). The grid reflects it on the next `_refresh_time` (see item 9 for the on-screen-refresh nuance).

**`finished` event survives session deletion:** `delete_session` ([db.py:896-912](src/fabulor/db.py#L896-L912)) touches **only** `listening_sessions` and `streak_grid_cache` — it never deletes from `book_events`. The `finished` event lives in `book_events` (`event_type='finished'`), written by the EOF flow ([app.py:1436](src/fabulor/app.py#L1436)) and only ever removed by `unfinish_book` / `delete_book_stats` / `reset_stats`. So a book's finished status does **not** depend on session-row existence: deleting every session for a finished book leaves its `book_events` finished row intact, and `has_finished_books`/`get_finished_book_data`/`get_streak_grid_finished_dates` (all keyed on `book_events`) still report it finished. The streak grid's finished-dot overlay (`get_streak_grid_finished_dates` [db.py:1044-1054](src/fabulor/db.py#L1044-L1054)) is sourced from `book_events.event_time` independently of the listened-cell cache, so a deleted session removes the listened fill but keeps the finished dot — correct and intentional. ✅

## 8. Checkpoint-recovered session contributes to streak grid — (a) correct

`_recover_checkpoint` ([session_recorder.py:210-253](src/fabulor/session_recorder.py#L210-L253)) writes the recovered session via the **same** `self._db.write_session(...)` call ([session_recorder.py:230-243](src/fabulor/session_recorder.py#L230-L243)) used by a normal `close()`, threading `day_start_hour=self._get_day_start_hour()` ([session_recorder.py:226](src/fabulor/session_recorder.py#L226), [session_recorder.py:242](src/fabulor/session_recorder.py#L242)). `write_session` ([db.py:509-547](src/fabulor/db.py#L509-L547)) unconditionally updates the streak grid cell(s) for the session's adjusted start/end dates ([db.py:540-547](src/fabulor/db.py#L540-L547)) — there is **no** "session completed normally" flag, no `is_crash_recovered` column, and **no `WHERE` clause anywhere** that distinguishes a recovered row from a normally-closed one. The `listening_sessions` schema ([db.py:73-87](src/fabulor/db.py#L73-L87)) has no such discriminator column. So a checkpoint-recovered session is byte-identical (in schema terms) to any other and is picked up by every streak query (`get_active_periods`, `build_streak_grid_cache`, `get_streak_grid_cache`) exactly the same way.

The only gate is the 60s threshold check in `_recover_checkpoint` itself ([session_recorder.py:215-218](src/fabulor/session_recorder.py#L215-L218)): if `listened < 60` the checkpoint is discarded without writing — matching the same `>= 60` rule that `close()` applies ([session_recorder.py:111](src/fabulor/session_recorder.py#L111)), so it is consistent, not an exclusion. The recovered session contributes to the streak grid for the day the crash occurred (the `session_start` carried in the checkpoint preserves the original date). ✅

## 9. History widget ↔ streak grid: independent queries, no shared stale cache — (a)* correct (DB), with a UI-staleness nuance

**Independent queries, no shared mutable cache:** the two surfaces read different sources with no in-memory cache bridging them:
- `_RecentHistoryWidget` / `_HistoryRow` (book detail panel) ← `get_book_sessions(book_id)` ([book_detail_panel.py:974](src/fabulor/ui/book_detail_panel.py#L974), [db.py:878-894](src/fabulor/db.py#L878-L894)).
- `StreakGrid` (stats panel) ← `get_streak_grid_cache()` ([stats_panel.py:2615](src/fabulor/ui/stats_panel.py#L2615)).

There is no shared Python object/cache between them — `StreakGrid._cache` ([stats_panel.py:1377](src/fabulor/ui/stats_panel.py#L1377)) is a private copy of the DB cache table, and `_RecentHistoryWidget` holds only its own `_history_rows`. So neither can read the other's stale in-memory state. ✅ for the literal check.

**Nuance (the medium finding):** the DB is always consistent after `delete_session` (item 5), but the *already-rendered stats-panel `StreakGrid` widget* can show a stale cell. Trace:
1. User opens the book detail panel **from a stats row** — `_on_book_row_clicked → open_book_detail(..., tab='stats')` ([stats_panel.py:2697-2699](src/fabulor/ui/stats_panel.py#L2697-L2699)).
2. `open_book_detail` slides the detail panel in **over** the stats panel ([panels.py:454-457](src/fabulor/ui/panels.py#L454-L457)) — it does **not** hide the stats panel, so the stats `StreakGrid` stays mounted with its last-rendered cells underneath.
3. User deletes a session in the detail panel's History tab → `delete_session` runs (DB + cache correct), then `_on_history_delete_confirmed → _finish → _refresh_stats()` ([book_detail_panel.py:1093-1098](src/fabulor/ui/book_detail_panel.py#L1093-L1098)) — which refreshes the *book detail panel's* stats, **not** the stats panel's streak grid.
4. User closes the detail panel → `_on_book_detail_hidden` ([panels.py:472-481](src/fabulor/ui/panels.py#L472-L481)) just hides it + `_notify_panel_closed()`; the stats panel's `showEvent` ([stats_panel.py:2554-2556](src/fabulor/ui/stats_panel.py#L2554-L2556)) installs an event filter but **does not re-query**. There is no refresh-on-reveal for the stats panel.

So if the deleted session was the day's only session, the underlying stats `StreakGrid` keeps showing that day filled until the user changes tabs (`_on_tab_changed → _refresh_time` [stats_panel.py:2511](src/fabulor/ui/stats_panel.py#L2511)) or the panel is otherwise refreshed. `delete_session` does **not** emit `session_written` (only `close()` does, [session_recorder.py:145](src/fabulor/session_recorder.py#L145)), so the `_on_session_written → refresh_current_tab` path ([app.py:871-873](src/fabulor/app.py#L871-L873)) is **not** triggered by a deletion. This is a **display-staleness** issue, not a data-correctness one — the cell is correct in the DB and corrects on the next refresh. Flagged medium and detailed in Findings.

## Findings requiring follow-up

Notes only — no fixes applied (audit-only task). No `(b)` data-correctness violations.

1. **[medium] Stats-panel `StreakGrid` can render a stale cell after a per-session delete (item 9).** The DB streak cache is invalidated correctly inside `delete_session`'s transaction, but the already-mounted stats-panel grid (left visible underneath the book-detail overlay) is not re-queried on detail-panel close. It self-corrects on the next tab change or stats refresh. Reachable path: open book detail from a stats row → delete the day's last session → close back to the Timeline tab. Candidate levers (not applied): (a) have `delete_session`'s caller, or `_on_history_delete_confirmed`, signal the stats panel to `_refresh_time()` (e.g. reuse/emit `session_written`, or a new `session_deleted` signal); or (b) add a refresh-on-reveal to the stats panel's `showEvent`/`_on_book_detail_hidden` path so returning to a visible Timeline tab re-queries. Option (a) is narrower. The book detail panel's own History/stats refresh after delete is already correct ([book_detail_panel.py:1096](src/fabulor/ui/book_detail_panel.py#L1096)); only the *other* surface lags.

2. **[informational] `delete_session` does not emit `session_written` (item 9).** Intentional today (the signal name means "a session was persisted," not "session data changed"), but it is the reason the stats-panel refresh fan-out doesn't fire on deletion. If finding #1 is addressed via a signal, prefer a distinct `session_deleted` rather than overloading `session_written`, to keep the "written" semantics clean for any future consumer that means it literally.
