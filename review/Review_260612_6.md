## 1. VT stop-and-load: five ANDed clauses — (a) correct

The VT same-file stop-and-load trigger is the single `if` at [player.py:552-556](src/fabulor/player.py#L552-L556):
```python
if (target_file['file_path'].lower().endswith('.mp3')                                   # (i)  .mp3
        and abs(local_pos - ((self._cached_time_pos or 0.0) - self._file_offset)) > _MP3_SEEK_THRESHOLD  # (ii) distance > 60s
        and os.path.getsize(target_file['file_path']) > _VT_MP3_SIZE_THRESHOLD          # (iii) file > 40MB
        and 2.0 < local_pos < target_file['duration'] - 5.0                              # (iv) local_pos in (2.0, dur-5.0)
        and not self._mp3_seek_reload_pending):                                          # (v)  not reload-pending
    self._mp3_stop_and_load(pos, file_path=target_file['file_path'], local_pos=local_pos)
    return
```
All five are joined with `and` in one boolean expression — there is **no** second VT call site that can reach `_mp3_stop_and_load` with a missing clause. (The only other caller of `_mp3_stop_and_load` is the **non-VT** branch at [player.py:575-579](src/fabulor/player.py#L575-L579), which is a different code path with four clauses — `.mp3` / distance>60s / not-reload-pending / single-file via `_play_target` — and is out of scope for the *VT* condition. Its narrower guard is correct for single-file books: `_VT_MP3_SIZE_THRESHOLD` and the `2.0 < local_pos < dur-5.0` bound don't apply because `_mp3_stop_and_load` itself has the 5s EOF buffer baked in via `start=` positioning, and single-file MP3 always uses the Xing/TOC header.)

Note the audit item phrases clause (iv) as `local_pos < file_duration`; the implementation is **stricter** — `2.0 < local_pos < target_file['duration'] - 5.0` — which both excludes near-start seeks (where stream-scan is cheap) and keeps a 5s EOF buffer. Stricter than stated, so no clause is missing. ✅

`os.path.getsize` ([player.py:554](src/fabulor/player.py#L554)) is a synchronous stat on the seek path; acceptable for a local file and unchanged from prior passes — noted, not flagged.

## 2. `handle_forward`/`handle_rewind` guards — (a)* correct (drop-not-queue is intentional)

The audit item frames this as a single compound guard `reload_pending AND old_pos is None`; in the code these are **two independent, sequential guards** with the same effect at both handlers:

`handle_rewind` ([app.py:2128-2142](src/fabulor/app.py#L2128-L2142)):
```python
if self.player and not self.player.mp3_seek_reload_pending:   # guard A
    old_pos = self.player.time_pos
    if old_pos is None:                                        # guard B
        return
    ...
```
`handle_forward` ([app.py:2144-2158](src/fabulor/app.py#L2144-L2158)):
```python
if self.player and not self.player.eof_reached and not self.player.mp3_seek_reload_pending:   # guard A (+ eof)
    old_pos = self.player.time_pos
    if old_pos is None:                                        # guard B
        return
    ...
```

**Guard A (`not mp3_seek_reload_pending`)** is identical in both; `handle_forward` additionally ANDs `not eof_reached` (correct — you can't skip forward past EOF). **Guard B (`old_pos is None` → return)** is byte-identical in both. So the two handlers apply the reload-pending and null-position guards identically.

**Behavior when `reload_pending is True` and `old_pos is not None`:** guard A is False, so the **entire handler body is skipped** — the skip request is **dropped, not queued, not applied**. This is the intended behavior: a stop-and-load reload is a `loadfile`-in-flight window (typically <200ms); issuing a second seek/skip into a reloading stream is exactly the "stacked loadfile" hazard the CLAUDE.md MP3-seek rules guard against (a second `_on_file_loaded` would bypass the early-return and re-emit `book_ready`). Dropping is correct and matches intent. The only observable effect is that a skip pressed during the brief reload window is a no-op; the user re-presses. No ambiguity, no queue needed.

✅* — `*` only because the audit's "compound guard" framing differs from the code's two-guard structure; functionally correct and consistent across both handlers.

- [player.py:553](src/fabulor/player.py#L553) stop-and-load distance check: `(self._cached_time_pos or 0.0) - self._file_offset` — **subtracts** the offset to recover the previous local position for comparison against the new `local_pos`. This is the inverse operation and is correct: it compares local-to-local. ✅

No call site adds `_file_offset` to a value already returned by the `time_pos` getter (which has the offset baked in). The getter is the sole translation point and is applied exactly once per logical read. ✅

The thread-hop to the Qt main thread happens at the **signal-emission** boundary: each callback only mutates plain attributes (`_cached_*`) and emits Qt signals, and **those signals are connected with `QueuedConnection`** in app.py:
- `chapter_changed.connect(self._update_chapter_label_from_index, Qt.ConnectionType.QueuedConnection)` ([app.py:337](src/fabulor/app.py#L337))
- `book_ready.connect(self._on_file_ready, Qt.ConnectionType.QueuedConnection)` ([app.py:338](src/fabulor/app.py#L338))
- `book_ready.connect(self._on_file_loaded_populate_chapters, Qt.ConnectionType.QueuedConnection)` ([app.py:339](src/fabulor/app.py#L339))
- `file_switched.connect(self._on_vt_file_switched, Qt.ConnectionType.QueuedConnection)` ([app.py:340](src/fabulor/app.py#L340))
- `load_failed.connect(self._on_load_failed, Qt.ConnectionType.QueuedConnection)` ([app.py:341](src/fabulor/app.py#L341))

So the "mpv C-thread callback → Qt slot must be queued" rule is satisfied **correctly and at the right layer**: the callbacks themselves do only thread-safe cached-attr writes and signal emits; every signal that lands on a Qt slot is `QueuedConnection`. The hot reads (`time_pos`/`duration`/`pause`/`speed` getters) read the `_cached_*` attributes the callbacks set — they never call into mpv on the Qt thread. ✅‡

(`_on_pause_test` at [player.py:177-185](src/fabulor/player.py#L177-L185) does read `self.instance.time_pos`/`.duration` directly inside the C-thread callback for EOF detection — this is on mpv's own thread reading mpv's own state, not a cross-thread Qt access, so it does not violate the rule.)

## 6. Findings — VT→M4B chapter-stuck-after-VT-session (trace only, NOT fixed)

**Goal:** trace VT session end → next M4B load → chapter-tracking init, listing every place `_chapter_list`, `_virtual_timeline`, `_last_nonvt_chapter` (and `_last_vt_chapter`) are reset or **not** reset across the transition, and flag any state that survives when it shouldn't.

**Transition path:**

1. **VT session running.** State: `_virtual_timeline = [...]` (non-None), `_chapter_list = [...]` (VT chapter data), `_last_vt_chapter` tracking the current VT chapter, `_file_offset` = current file's `cumulative_start`, `_current_vt_index` ≥ 0.

2. **User selects an M4B book → `load_book(path)` ([player.py:311](src/fabulor/player.py#L311)).** The reset block ([player.py:343-360](src/fabulor/player.py#L343-L360)) runs synchronously **before** the async `_ResolveWorker` is queued. Every relevant field is cleared:
   - `_virtual_timeline = None` ([player.py:344](src/fabulor/player.py#L344)) ✅ reset
   - `_file_offset = 0.0` ([player.py:345](src/fabulor/player.py#L345)) ✅ reset
   - `_chapter_list = None` ([player.py:347](src/fabulor/player.py#L347)) ✅ reset
   - `_current_vt_index = 0` ([player.py:348](src/fabulor/player.py#L348)) ✅ reset
   - `_last_vt_chapter = -1` ([player.py:351](src/fabulor/player.py#L351)) ✅ reset
   - `_last_nonvt_chapter = -1` ([player.py:352](src/fabulor/player.py#L352)) ✅ reset
   - `_cached_time_pos = None`, `_cached_duration = None`, `_seek_target = None` ([player.py:358-360](src/fabulor/player.py#L358-L360)) ✅ reset
   - `_is_seeking = True` is set at [player.py:313](src/fabulor/player.py#L313).

   **So: no VT/chapter tracking state survives `load_book` into the M4B session.** `_last_nonvt_chapter` starts at -1, which forces the first `_on_time_pos_change` walk to emit (since any computed `curr` ≠ -1).

3. **`_resolve_playlist` for a single M4B ([player.py:196-209](src/fabulor/player.py#L196-L209)).** With `chapter_source == 'embedded'` (default), it does **not** set `_chapter_list` (stays None) and does **not** set `_virtual_timeline` (stays None). Returns `(str(audio_file), None)`. The M4B's chapters come from mpv's native `instance.chapter_list` via the `chapter_list` property fallback ([player.py:636-639 region — getter at lines 624-627](src/fabulor/player.py#L624-L627)).

4. **`_on_file_loaded` ([player.py:414](src/fabulor/player.py#L414)) → `book_ready.emit()`** (non-VT branch, [player.py:431](src/fabulor/player.py#L431); after the new pending-seek guard which is a no-op here since `_pending_local_pos` was reset to None at [player.py:349](src/fabulor/player.py#L349)).

5. **Chapter tracking init for the M4B.** `_on_time_pos_change` ([player.py:110](src/fabulor/player.py#L110)) takes the **non-VT branch** ([player.py:141-148](src/fabulor/player.py#L141-L148)) because `_virtual_timeline is None`. It walks `self.chapter_list` (mpv native) against `value + _CHAPTER_BOUNDARY_EPSILON` and emits when `curr != _last_nonvt_chapter` (which is -1, so the first valid walk emits).

**Verdict of the trace:** **No VT state survives the VT→M4B transition.** `load_book`'s reset block is comprehensive and runs before any new-book work. The symptom ("M4B chapter display freezes at a chapter boundary in some M4B books" — CLAUDE.md Pending/Known Debt) is therefore **not** explained by leaked `_virtual_timeline`/`_chapter_list`/`_last_nonvt_chapter` across the switch.

**Where the freeze more likely originates (candidates for a future investigation, not asserted):**
- **mpv native `chapter_list` timing.** For embedded M4B the walk depends on `self.instance.chapter_list` being populated. If `_on_time_pos_change` fires before mpv has parsed the chapter table, `self.chapter_list` is briefly `[]` and the branch is skipped (`elif self.chapter_list and value is not None` is False) — no emit until chapters arrive. If a particular M4B reports `time-pos` updates but a malformed/empty native chapter list, the non-VT branch never advances `_last_nonvt_chapter` past its initial computed value → the label "sticks." This is an mpv-data property, not a Fabulor state-leak.
- **The settle-reset only fires inside the seek-clear branch** ([player.py:115-124](src/fabulor/player.py#L115-L124)): `_last_nonvt_chapter = -1` is re-asserted only when `_is_seeking` clears with `_seek_target is not None`. On a normal play-through (no seek) across a chapter boundary, the advance relies solely on `curr != _last_nonvt_chapter` in the walk — correct, but if two boundaries are crossed within one suppressed window (e.g. during the `is_seeking` gate after the M4B's restore seek), only the final chapter emits. Not a leak; a coalescing property of the gate.

**Flagged state that survives but should be benign:** none that is incorrect. `_mp3_seek_*` flags are reset ([player.py:354-355](src/fabulor/player.py#L354-L355)); `_undo_pos` and `_paused_time` (app-side) are not reset in `load_book` but are unrelated to chapter tracking. No chapter-tracking state crosses the boundary.

## 7. Findings — Progress-slider race on book switch (trace only, NOT fixed)

So the guard holds **as long as `_flow_anim` is still Running.** The thin window is: flow animation finished (guard A False) AND seek settled (guard B False) AND the live `pos` transiently differs from `new_val`. Low-probability, self-correcting on the next tick.

The disconnect-before-connect fix noted in CLAUDE.md (`load_book` re-wiring) addresses the *double-handler* variant; this trace did not find a *second* unguarded synchronous writer — the exposure is purely the timing overlap of two guards releasing in the wrong order, not a missing guard.
