## 1. Sidebar dispatch guard — Variant A confirmed, (a)

**Variant A (centralized dispatch guard) is what is implemented**, not a partial/alternate approach. The re-arm guard lives in the single shared handler [`_on_sidebar_closed_for_panel`](src/fabulor/ui/panels.py#L295-L352), not scattered across the six `_open_*_flow` call sites:

```python
if self.sidebar_expanded:            # panels.py:328
    # dropped close, or this `finished` belonged to an OPENING animation
    self._toggle_sidebar()           # re-issue the close
    return                           # do NOT dispatch, do NOT disconnect
```

The dispatch (the `if self._pending_panel_open == ...: _start_*_entry()` ladder, [panels.py:342-347](src/fabulor/ui/panels.py#L342-L347)) is reached only after the `sidebar_expanded == False` gate. All six `_open_*_flow` methods ([library:89](src/fabulor/ui/panels.py#L89), [settings:173](src/fabulor/ui/panels.py#L173), [speed:264](src/fabulor/ui/panels.py#L264), [stats:419](src/fabulor/ui/panels.py#L419), [sleep:450](src/fabulor/ui/panels.py#L450), [tags:529](src/fabulor/ui/panels.py#L529)) funnel through this one handler via `sidebar_animation.finished` — the guard is genuinely centralized.

**No other call site dispatches a panel-open past the re-armed guard.** The only two non-`_open_*_flow` `_toggle_sidebar` callers are:
- [`hide_all_panels`:706](src/fabulor/ui/panels.py#L706) — a plain close (`if self.sidebar_expanded: self._toggle_sidebar()`), no `_pending_panel_open`, no dispatch.
- [`handle_drag_area_right_click`:773](src/fabulor/ui/panels.py#L773) — the "no panel visible" branch, a bare sidebar toggle, no dispatch.

Neither queues or dispatches a panel, so neither can bleak a panel over an expanded sidebar.

**Termination is sound.** `finished` fires only when the animation reaches Stopped, so the re-issued `_toggle_sidebar()` at [panels.py:335](src/fabulor/ui/panels.py#L335) is *not* subject to the `state() == Running` no-op at [panels.py:53-54](src/fabulor/ui/panels.py#L53-L54) — it always starts a real close. `sidebar_expanded` can only flip back to `True` via a user-driven opening toggle, so the re-arm cannot self-perpetuate. The docstring's termination argument matches the code.

**Settings-panel first-appearance transient (noted, not fixed — per prompt).** `_open_settings_flow` is the one flow that calls `theme_manager.snap_theme_forward()` *before* the sidebar-expanded branch ([panels.py:175](src/fabulor/ui/panels.py#L175)), unlike the other five. This is orthogonal to the re-arm guard (it runs on every settings open regardless of sidebar state) and is a plausible surface for a first-open visual transient, but it is not a guard defect and was out of scope to fix. Recording it here as the requested "note, do not fix."

## 3. Scan/missing-book teardown — **(b), HIGH — ESCALATED**

**The currently-loaded-book-flagged-missing check queries the wrong flag and no longer fires for the case it exists for.**

[library_controller.py:134-142](src/fabulor/library_controller.py#L134-L142):

```python
current = self.app.get_current_file()
if current and self.db.is_book_excluded(current):     # ← is_book_excluded, NOT is_book_missing
    self.app.on_book_removed()
    ...
    return
```

The handler was written (commit `003c752`, 2026-06-26) *when force-rescan missing-detection wrote `is_excluded=1`*. On 2026-06-27 the `is_missing`/`is_excluded` split (commits `98def39` → `9afab19`) moved missing-detection to write **`is_missing=1` only** — `mark_books_missing` ([db.py:1633-1646](src/fabulor/db.py#L1633-L1646)) never touches `is_excluded`. The comment above the check *was* updated in the same window to reference "the is_missing/is_excluded split" ([library_controller.py:126-133](src/fabulor/library_controller.py#L126-L133)), but **the predicate was not** — it still calls `is_book_excluded` ([db.py:1582-1587](src/fabulor/db.py#L1582): `SELECT is_excluded ...`), while the matching `is_book_missing` ([db.py:1589-1594](src/fabulor/db.py#L1589)) exists and is not called.

**Failure scenario (concrete):** User is playing book B. Its folder is deleted from disk. User hits Rescan (force). The scanner flags B `is_missing=1` (`is_excluded` stays `0`). `_on_scan_finished` runs → `is_book_excluded(B)` returns `False` → the teardown branch is skipped → **B stays open in the player** (ghost now-playing state, session not closed, player not torn down) until the user manually tries to reload it. That later reload is caught by the *lazy* single-book detector ([app.py:1067-1076](src/fabulor/app.py#L1067)), which does correctly call `_on_book_removed()` — so the state is eventually reachable, but the scanner-path teardown this handler was added to provide is silently dead.

The branch *does* still fire in one corner: a book that was already user-trashed (`is_excluded=1`) before going missing — the "accepted edge case" from CLAUDE.md — which is the minority path, not the primary one the handler targets.

**Not a double-fire / race.** `_on_scan_finished` is connected exactly once ([app.py:403](src/fabulor/app.py#L403)); `on_book_removed()` is called once inside the guarded branch; there is no competing scan-completion slot calling it. The *mechanism* is clean — the sole defect is the wrong predicate.

**Escalation basis:** the docstring/comment was updated to assert the split is handled while the code was not — a commit's stated behavior diverging from its diff. Per the prompt this stops-and-reports. **No fix applied** (audit-only). The one-line nature (`is_book_excluded` → an OR with `is_book_missing`, or a dedicated "not visible" predicate) is noted for Prompt 2 / a follow-up fix task, not performed here.
