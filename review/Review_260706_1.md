## Summary of what the premise got right and wrong

The prompt's "what we already know" section supplied three claims to verify. Verified outcome for each:

| Claim | Verdict | Detail |
|---|---|---|
| `q` is handled in `ChapterList.mousePressEvent` | **WRONG** | `Q` (and `Escape`, `C`) are handled in `ChapterList.keyPressEvent` ([chapter_list.py:277-294](src/fabulor/ui/chapter_list.py#L277-L294)). `ChapterList.mousePressEvent` ([chapter_list.py:266-275](src/fabulor/ui/chapter_list.py#L266-L275)) handles left/right mouse clicks on list rows — a completely different concern (row activation, not dismissal). |
| `t`/`c` "possibly also" wired like `q` in mouse handlers | **WRONG** | `T` and `C` (global, app-wide) are handled in `MainWindow.keyPressEvent` ([app.py:2109-2123](src/fabulor/app.py#L2109-L2123)) — a real `keyPressEvent` if/elif chain, not mouse-event-based. Separately, `ChapterList` *also* has its own `C` handling in its own `keyPressEvent` (see collision #1 below) — two different `C` handlers in two different classes. |
| There's a `keyPressEvent` if/elif chain for at least some of q/t/c, but the picture may be messier (mixed mechanisms) | **PARTIALLY RIGHT, for the wrong reason** | There genuinely are two separate `keyPressEvent` if/elif chains — `MainWindow.keyPressEvent` (app.py:2109) and `ChapterList.keyPressEvent` (chapter_list.py:277) — but neither is mouse-event-based. The "messier than expected" instinct was correct; the specific mechanism (mouse events) was not. Every single-key binding found is either a `keyPressEvent` override or a monkey-patched-per-widget `keyPressEvent`/`focusInEvent`; **zero** use `QShortcut`, `setShortcut()`, `QAction`, or a shared event filter for actual key dispatch. |

## 2. Arrow-key chapter-list navigation — resolved

**"Opening from the main window" (the `C` key) and "opening by clicking" (the chapter label) are the *same* code path, not two.** There is exactly one `ChapterList` instance ([app.py:536](src/fabulor/app.py#L536): `self.chapter_list_widget = ChapterList(self)`) and exactly one call site for `show_above` ([app.py:1677](src/fabulor/app.py#L1677), inside `_show_chapter_dropdown`). Both the `C` keypress ([app.py:2111](src/fabulor/app.py#L2111)) and the label's `clicked` signal ([main_window_builders.py:328](src/fabulor/ui/main_window_builders.py#L328)) call `self._show_chapter_dropdown()` directly — there is no branch, wrapper, or divergent path between them. `_show_chapter_dropdown` ([app.py:1644-1681](src/fabulor/app.py#L1644-L1681)) also handles the "already open → close it" toggle itself (line 1649: `if self.chapter_list_widget.isVisible(): ... fade_out(); return`), so both triggers get the open/close toggle behavior identically.

**Arrow-key scoping mechanism: genuine Qt keyboard focus, not a flag or modal dialog.** `ChapterList` is a real `QListWidget` subclass (not a `QDialog`, no modal event loop) with `setWindowFlags(Qt.Widget)` ([chapter_list.py:75](src/fabulor/ui/chapter_list.py#L75)) — it's a plain child widget of `MainWindow`, positioned absolutely and shown/hidden via opacity animation, not Qt's window-stacking. It never explicitly sets a `focusPolicy`, so it inherits `QListWidget`'s Qt-native default (`Qt.StrongFocus` — accepts focus via click, tab, or programmatic `setFocus()`). Focus is acquired **programmatically**: `QTimer.singleShot(0, self.setFocus)` is called both in `show_above` ([chapter_list.py:197](src/fabulor/ui/chapter_list.py#L197), on open) and in `_toggle_expand` ([chapter_list.py:227](src/fabulor/ui/chapter_list.py#L227), on expand/collapse). Once focused, Qt's normal event delivery routes `Up`/`Down`/`Left`/`Right`/digits/`Enter`/`Escape` to `ChapterList.keyPressEvent` before `MainWindow.keyPressEvent` ever sees them (a focused child widget gets first refusal on key events in Qt's propagation model) — there is no manual boolean gate like `_chapter_ui_active` involved in *this* particular scoping; that flag (from CLAUDE.md) governs slider cursor/stylesheet/mouse-transparency state, a different concern from keyboard focus.

**One second Qt focus consumer to note:** the digit-buffer timeout (`_digit_timer`, 800ms) is independent of focus — it's a plain `QTimer`, so a rapid digit sequence commits even if focus is somehow lost mid-sequence before the timer fires (not observed as a bug, just a note on mechanism).

## 3. Existing shortcut-adjacent infrastructure

- **No dedicated shortcuts/actions/constants module exists.** `find src -iname "*constant*" -o -iname "*enum*" -o -iname "*action*"` returns nothing.
- **No `QShortcut`/`QAction` precedent** to migrate away from (§1e) — a new module starts from zero, not from a partial pattern.
- **Config/settings scaffolding pattern that a future configurable-bindings store would likely follow:** `Config` ([config.py:3](src/fabulor/config.py#L3)) wraps `QSettings("Fabulor", "Fabulor")` with simple `get_x()`/`set_x()` pairs per setting, string- or bool-valued, e.g. `get_scroll_mode()`/`set_scroll_mode()` ([config.py:183-187](src/fabulor/config.py#L183-L187), values like `"Slow"`/`"Normal"`/`"Off"`), `get_chapter_digit_mode()`/`set_chapter_digit_mode()` ([config.py:267-271](src/fabulor/config.py#L267-L271), `"by_name"`/`"by_index"`). No existing setting stores a *key code* — all existing "mode" settings are closed-enum string choices, not open key-capture values. A future "configurable key for action X" setting would be a new shape for this file, not a drop-in extension of the existing pattern.
- **One enum-based state-machine precedent worth knowing about (not shortcut-related, but a local convention for state):** `_MetaActionState(Enum)` in `book_detail_panel.py:23` (`from enum import Enum, auto`). This is the only `Enum`-based state pattern found in the UI layer — relevant only as a "how does this codebase model discrete states" data point, since a shortcuts module will likely need something like an `Action` enum.
- **Settings UI precedent for exposing a discrete-choice setting to the user:** the Controls tab already exposes `chapter_digit_mode` (by_name/by_index) and `chapter_digit_autoplay` as clickable-button toggles wired via `main_window_builders.py:1016-1024`, emitting Qt signals (`chapter_digit_mode_changed`, etc.) that `SettingsController` picks up. This is the existing "expose a mode in Settings" pattern the prompt's eventual configurable-bindings goal would likely extend, but it has no existing analog for capturing an arbitrary key (all current toggles are fixed 2-3-way choices via buttons, not a "press a key to bind" capture flow).

## 4. Collisions and oddities

1. **Two independent `C` handlers in two different classes, both reachable while the dropdown is open.** `MainWindow.keyPressEvent` ([app.py:2110](src/fabulor/app.py#L2110)) opens the dropdown on `C` (gated on `_chapter_label_clickable`). `ChapterList.keyPressEvent` ([chapter_list.py:287](src/fabulor/ui/chapter_list.py#L287)) *closes* the dropdown on `C` (grouped with `Escape`). This is not a bug today — because `ChapterList` steals Qt focus while open, `MainWindow.keyPressEvent` never sees the `C` keypress in that state, so there's no double-fire — but it is two different classes independently agreeing that `C` means "toggle this overlay," coordinated only by Qt's focus-delivery order rather than by any explicit shared constant or handshake. A future shortcuts module that centralizes key→action mapping will need to decide whether this dual-registration (open in one class, close in another) collapses into one binding or stays deliberately split by context.

2. **`Q` is simultaneously "quote rotate" (global, `MainWindow`) and "row activate as force-play... no wait, that's `Space`."** Correction while writing this: `Q` has only the one meaning (quote rotation, `MainWindow`-level, testing-only). Flagging that it's explicitly commented `# TODO: remove before release — testing only` ([app.py:2119](src/fabulor/app.py#L2119)) — worth deciding during design whether `Q` gets freed up for a real binding or stays reserved/removed, since it's presently dead-weight-adjacent (only active in the empty/no-book state).

3. **The upcoming new `l` shortcut has no naming/case collision today** — no existing binding uses `L` (upper or lower) at any scope found in this sweep. Clear to bind.

4. **`Escape` is independently reimplemented four times** (§1c) with no shared helper — each widget's monkey-patched handler duplicates "clear + clearFocus" or "revert + clearFocus" logic inline. Not a functional bug (each is correctly scoped to its own widget), but a design-relevant data point: a shortcuts module aiming to centralize dispatch will need to decide whether these local, non-global, widget-scoped `Escape` handlers are in scope for the new system at all, or intentionally left alone as local text-field UX rather than "app shortcuts." They don't compete with each other (mutually exclusive by focus) or with the global `Q`/`T`/`C` chain (`MainWindow.keyPressEvent`'s `else` branch calls `super()`, so it never intercepts a key a focused child already consumed).

5. **`BookDetailPanel`'s `Escape` handler is installed on `QApplication` (app-wide) but logically self-gates on `self._editing`.** ([book_detail_panel.py:923](src/fabulor/ui/book_detail_panel.py#L923), [book_detail_panel.py:957-960](src/fabulor/ui/book_detail_panel.py#L957-L960)) This is a *broader-than-necessary* filter scope (sees every keypress in the whole app) compared to `TagManagerWidget`'s narrower `obj is self._tag_name_edit` check on the same `installEventFilter(self)` pattern. Not a bug (the `_editing` gate makes it inert otherwise), but a design-relevant inconsistency: two panels solve "scope Escape to my own editing state" with event filters of different installation breadth (`QApplication`-wide vs. object-identity-checked). A shortcuts module standardizing on one approach should pick one of these two shapes deliberately, not inherit both.

6. **No dead code found among the key handlers themselves.** Every `keyPressEvent`/`mousePressEvent`/`eventFilter` branch inventoried above is reachable and does something (including the `Q` testing-only branch, which is reachable in the empty-library state, not orphaned).

## Open questions for design

These are flagged as things the `shortcuts.py` architecture should resolve *explicitly*, not assume:

1. **Does the new system centralize the `ChapterList`-scoped keys (§1b) or leave them alone?** They're cleanly scoped today via genuine Qt focus on a real widget — arguably already "correct" and possibly out of scope for a global-shortcuts module, which more naturally targets `MainWindow`-level bindings like `C`/`T`/`Q` (§1a). Needs an explicit decision, not a default assumption that "all keys" means all of §1a *and* §1b *and* §1c.

2. **Are the widget-scoped `Escape`-to-cancel handlers (§1c) in scope?** They're local text-field UX, each already correctly isolated by focus/object-identity. Folding them into a central dispatcher risks over-engineering a set of handlers that don't currently collide with anything and don't need Settings-configurability (no user would want to rebind "Escape clears the search box").

3. **Does "Settings-configurable" apply only to the `MainWindow`-level global keys (`T`/`C`/possibly the new `l`), or to the `ChapterList` in-overlay keys too (`Left`/`Right`/digit-jump mode already has a config toggle, `chapter_digit_mode`, but the *keys themselves* — `0`-`9`, arrows — aren't currently rebindable, just their *interpretation* is)?** This distinction (rebinding a trigger key vs. configuring what a fixed trigger does) already exists once in the codebase (`chapter_digit_mode`) and the new design should be clear about which of the two it's building.

4. **What happens to `Q`?** It's explicitly marked testing-only and scoped to a state (empty library, no book) that's orthogonal to every other binding. Decide during design whether it's removed, repurposed, or left alone as pre-existing debt outside this effort's scope.

5. **Should the new system also absorb wheel-based "shortcuts" (§1d) for architectural consistency, or are those explicitly out of scope as mouse/analog input rather than discrete key bindings?** They're functionally similar (context-dependent input → app action) but structurally very different (continuous delta vs. discrete keypress) and already centralized in one `wheelEvent` override — worth an explicit call on whether "shortcuts" as a concept includes them.

6. **`MainWindow.keyPressEvent`'s reach given Qt's focus-delivery order.** The report above states that a focused child widget (e.g., an open `QLineEdit` in a panel) would consume relevant keys before `MainWindow.keyPressEvent` runs — this was reasoned from Qt's standard event-propagation model, not verified with a live/instrumented test in this pass (out of scope: audit-only, no code run). If the design phase needs a stronger guarantee (e.g., "does `T` in a `QLineEdit` fire the global theme-rotate handler or not"), that should be confirmed live before being relied upon.
