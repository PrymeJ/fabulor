## 1. `upsert_book` vs `upsert_books_batch` SQL identity — (a) correct

Diffed the two query strings character-by-character:
- `upsert_book` query: [db.py:282-296](src/fabulor/db.py#L282-L296)
- `upsert_books_batch` query: [db.py:303-317](src/fabulor/db.py#L303-L317)

The `INSERT INTO ... VALUES (...) ON CONFLICT(path) DO UPDATE SET ...` bodies are **byte-for-byte identical**: same column list, same `COALESCE(:progress, 0)` insert, same nine `DO UPDATE SET` clauses in the same order, same `is_deleted=0` / `is_excluded=0` resets. They differ only in execution:
- `upsert_book` → `conn.execute(query, cleaned)` ([db.py:298](src/fabulor/db.py#L298)) on a single dict
- `upsert_books_batch` → `conn.executemany(query, cleaned_list)` ([db.py:329](src/fabulor/db.py#L329)) over a list comprehension

The `cleaned`/`cleaned_list` parameter prep also matches (same nine keys, same `.strip()`-if-str normalization). In sync. ✅

## 2. Four `X_locked` CASE guards — (a) correct

All four present and identical in both upserts:
- `title=CASE WHEN books.title_locked THEN books.title ELSE excluded.title END` ([db.py:287](src/fabulor/db.py#L287) / [308](src/fabulor/db.py#L308))
- `author=CASE WHEN books.author_locked THEN books.author ELSE excluded.author END` ([db.py:288](src/fabulor/db.py#L288) / [309](src/fabulor/db.py#L309))
- `narrator=CASE WHEN books.narrator_locked THEN books.narrator ELSE COALESCE(NULLIF(excluded.narrator, ''), books.narrator) END` ([db.py:289](src/fabulor/db.py#L289) / [310](src/fabulor/db.py#L310))
- `year=CASE WHEN books.year_locked THEN books.year ELSE COALESCE(excluded.year, books.year) END` ([db.py:293](src/fabulor/db.py#L293) / [314](src/fabulor/db.py#L314))

Note: the guards use the truthy bare-column form `WHEN books.title_locked` rather than the `= 1` form quoted in CLAUDE.md. Semantically equivalent in SQLite (column is `INTEGER NOT NULL DEFAULT 0`; any non-zero is truthy). Not a defect — flagging only as a wording mismatch vs. the rule text. ✅

## 7. Raw `0.0` progress at upsert call sites — (a) correct

- **scanner.py**: the book metadata dict ([scanner.py:254-260](src/fabulor/library/scanner.py#L254-L260)) has **no `"progress"` key**. `upsert_book` does `book_data.get("progress")` → `None` → `COALESCE(:progress, 0)`. The Signal named `progress` ([scanner.py:263](src/fabulor/library/scanner.py#L263)) is a `(processed, total)` progress *bar* signal, unrelated to DB progress. No `0.0` passed.
- **library_controller.py**: grep for `upsert_book` / `progress=` returns no upsert call sites at all — the controller does not call the upserts directly.

No raw `0.0` reaches either upsert. ✅

## 8. `book_path` still written by write_session / write_book_event — (a)* correct, with a note

This is a **trick-question check**: CLAUDE.md's "DB migration status" explicitly states `write_session` / `write_book_event` *"still dual-write `book_path` + `book_id`"* and the columns are "not yet dropped — pending final column drop pass." So `book_path` *being written* is the documented current state, not a violation.

- `write_session` [db.py:520-539](src/fabulor/db.py#L520-L539): INSERT includes `book_path` and binds `str(book_path) if book_path else None` ([db.py:529](src/fabulor/db.py#L529)). Still written.
- `write_book_event` [db.py:850-855](src/fabulor/db.py#L850-L855): INSERT includes `book_path` and binds it ([db.py:855](src/fabulor/db.py#L855)). Still written.

**Severity low** — matches documented intent. The audit question framed `book_path` as "no longer written"; in fact it *is* still written by design (dual-write during the path→ID migration). No action needed unless the column-drop pass has been scheduled. By contrast, `set_started_at`/`get_book_started_at` ([db.py:584-602](src/fabulor/db.py#L584-L602)) are fully migrated to `book_id` with no `book_path` — consistent with the rule. The asymmetry is intentional and documented.
