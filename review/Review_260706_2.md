# INVESTIGATION — Panel/overlay concurrent-open collisions ("two panels overlap")

> _Audited against commit `d186f86` (HEAD of `feat/shortcuts-module`). file:line citations reflect that tree._

Date: 2026-07-06
**Investigation only — no fix applied, no policy chosen.** Pryme asked for the full collision
matrix before a fix is designed. This maps every panel/overlay open entry point, its cleanup
behaviour, and which pairs can end up visible together.

---

## Root cause in one sentence

There is **no single mutual-exclusion gate for "an overlay is opening."** Each open entry point
independently decides whether to clear the others first — and most don't — so whether two overlays
coexist depends entirely on *which two entry points* fire and *in what order*.

---

Nine things can occupy the main-window overlay space, in three mechanism families:

| # | Overlay | Open entry | Mechanism family |

| 2 | Settings panel | `_open_settings_flow` | Sidebar panel |

| 4 | Sleep panel | `_open_sleep_flow` | Sidebar panel |

## Per-entry-point cleanup behaviour (the asymmetry)

What each entry point does about *other already-open overlays* before opening its own:

| Entry point | Clears others first? | How |
|---|---|---|
| `_open_library_flow` (1) | **NO** | Queues behind sidebar if `sidebar_expanded`, else calls `_start_library_entry()` directly. Never checks panels 2–8. |
| `_open_settings_flow` (2) | **NO** | Same shape. `snap_theme_forward()` only. No panel check. |
| `_open_speed_flow` (3) | **NO** | Same. `_abort_theme_fade()` only. |
| `_open_sleep_flow` (4) | **NO** | Same. |
| `_open_stats_flow` (5) | **NO** | Same. |
| `_open_tags_flow` (6) | **NO** | Same. |
| `_show_chapter_dropdown` (8) | **YES** | Calls `self.panel_manager.hide_all_panels()` at [app.py:1662](src/fabulor/app.py#L1662) before showing — the ONLY open path that proactively clears everything else. |
| Drag-area left-click ([app.py:2393](src/fabulor/app.py#L2393)) | N/A (never opens a panel) | If a panel is visible → `hide_all_panels()`; elif current_file → `toggle_play_pause()` (which also `hide_all_panels()`). It's a close/playback toggle, not a panel opener. |
| Sidebar button clicks ([panels.py:43-49](src/fabulor/ui/panels.py#L43)) | inherit their `_open_*_flow` | e.g. `library_trigger_btn.clicked → _open_library_flow` — so **NO** (same as the flow). |
| New `L` shortcut (`_open_library_shortcut`, app.py) | partial | No-ops if `is_any_full_panel_visible()` OR the library is open OR empty state; otherwise `_open_library_flow`. So it *guards against* opening over another panel — but only from the L side. |

**The single load-bearing fact:** exactly one opener (the chapter dropdown, #8) calls
`hide_all_panels()`. Every sidebar panel opener (#1–6) opens blind. The new `L` shortcut is the only
sidebar-family opener with *any* pre-open guard, and it's one-directional (it checks before opening,
but nothing checks against *it*).

## Collision matrix

"Collide" = both overlays can end up simultaneously visible / overlapping. Order-dependence noted.
Derived from the cleanup table above; the timing-sensitive cells (marked ⚠) depend on animation
state at the moment of the second action and match Pryme's live repros.

| Second action ↓ / First action → | Library open | Settings/Speed/Sleep/Stats/Tags open | Chapter dropdown open | Sidebar open (no panel yet) |
|---|---|---|---|---|
| **Open Library** (`l`, sidebar btn) | — (L-shortcut no-ops; sidebar-btn re-enters) | **COLLIDE** ⚠ — library opens over the other panel; neither clears the other | no-op-ish: chapter list stays, library opens over it ⚠ | queued-open (correct) |
| **Open Settings/Speed/etc.** (sidebar btn) | **COLLIDE** ⚠ | **COLLIDE** ⚠ (two sidebar panels) | opens over chapter list ⚠ | queued-open (correct) |
| **Open Chapter dropdown** (`c`, label click) | `hide_all_panels()` closes library, but a **close-slide vs open-slide fight** if library is mid-open ⚠ | closes the panel (may fight mid-animation ⚠) | toggles closed (correct) | `hide_all_panels()` closes sidebar (correct-ish) |
| **Playback toggle** (drag-area click) | closes the panel (correct) | closes the panel (correct) | does NOT close the chapter list (chapter list isn't in the drag-area `is_any_panel_visible` path the same way) — **needs live confirm** | — |

### Mapping to Pryme's exact repros
- **`l` then `c` → both appear.** `l` opens library blind (no cleanup). `c` calls `hide_all_panels()`
  which starts a *close* on the library that is still mid-*open-slide* — the two animations fight and
  the visible result is overlap. **`c` then `l` → fine**, because `c` clears first, then `l` opens into
  a clean field and nothing re-triggers `c`. ✔ matches.
- **`l` then sidebar panel click → both.** Two sidebar-family openers, neither clears the other. ✔
- **`l` + playback(drag-area) click, either order → both.** The playback/speed entry doesn't
  serialize against a library open. ✔ (Exact "playback panel" identity needs confirming — see open
  questions; the drag-area LEFT-click is play/pause, not a panel, so "playback panel" likely means
  the **speed** panel via the speed button, OR the drag-area interaction opening the sidebar. This is
  the one repro I can't fully pin to code without confirming which control you clicked.)
  **RESOLVED:** the "playback panel" is opened by a **single left-click on the main window / drag
  area** ([app.py:2393](src/fabulor/app.py#L2393) `_on_drag_area_pressed`), NOT the speed button.
  `l` + that click in either order collides.
- **`l` then Settings → both, mid-slide.** **RESOLVED (no key involved):** the path is right-click
  to open the sidebar → click **Settings** → tap `L` **while the Settings panel is mid-slide**. So the
  collision window is specifically *during another panel's slide animation*: `L`'s
  `is_any_full_panel_visible()` guard reads `settings_panel.isVisible()` — but the panel is already
  `show()`n at the *start* of its slide, so the guard SHOULD catch it... which means the observed
  overlap implies either the guard check races the `.show()`, or `_open_library_flow` proceeds anyway.
  This is the key timing cell the fix must close: a guard on *visible* is not enough; it must also
  cover *animating*.

## Why the sidebar-queued path is NOT enough

The queued-open pattern (`_pending_panel_open` + `_on_sidebar_closed_for_panel`, [panels.py:295](src/fabulor/ui/panels.py#L295))
correctly serialises **sidebar-close → panel-open** when you go *through the sidebar*. But it only
guards the sidebar handoff. It does nothing when:
- the sidebar is already closed and two `_start_*_entry()` calls race (both open immediately), or
- the chapter dropdown (a non-sidebar mechanism) races a sidebar panel.

So the existing serialisation is real but scoped to one specific transition, not to "only one overlay
at a time" globally.

---

## Shape of a fix (NOT chosen — for the design step)

A single choke point every opener routes through: "ensure all *other* overlays are closed (and let
their close animations settle / be interrupted cleanly), then open the requested one." Candidates
differ mainly on the collision *policy* Pryme will pick (switch-to-new vs ignore-second) and on how
to sequence close-then-open so they don't visually fight mid-slide — which is the actual hard part
here, given the animation-guard rules in CLAUDE.md (the flow-animation / theme-fade / seek guards
must not be disturbed). The chapter dropdown's `hide_all_panels()`-then-open is a partial precedent,
but note it already exhibits the close-vs-open *fight* against a mid-open library — so "just call
`hide_all_panels()` everywhere" is necessary but not sufficient; the sequencing/interrupt story has
to be solved too.

---

## RESOLVED decisions (Pryme, 2026-07-06)

1. **Collision policy: IGNORE the second request.** No queueing, no switching. If a second
   open-action arrives while an overlay is already up or in transition, it is dropped.
2. **Drop scope: OPEN *or* ANIMATING.** Strictly one overlay at a time. A second open is dropped when
   any overlay is fully open-and-settled OR any open/close animation is in flight. (This is stricter
   than a visible-only guard — it must also cover the mid-slide window, which is the exact case the
   `l`-then-Settings-mid-slide repro exposes.)
3. **Shortcuts are main-window-exclusive today.** A shortcut key acts only when overlays are settled
   AND nothing is open. Pressing `L` while Stats is open does NOTHING right now.
4. **FUTURE (explicitly not this task):** "press `L` while in Stats → dismiss Stats, open Library"
   (switch-behaviour) is deferred. Do not build it now.
5. **"Playback panel" = drag-area left-click** ([app.py:2393](src/fabulor/app.py#L2393)); **"l then
   Settings" = sidebar-click path, collision during the slide.** Both resolved above.

## Still open (design-level, not blocking)

- **Book-detail panel (#7, slides from the right)** — distinct mechanism, opens from library/stats
  contexts. Not in the repros. Decide during design whether the exclusion gate covers it too (likely
  yes for consistency, but its right-slide + context-open flow needs a look).
- **Where the single gate lives.** The natural choke point is a `PanelManager` predicate (e.g.
  `can_open_overlay()` = `not is_any_overlay_open_or_animating()`) that every `_open_*_flow`,
  `_show_chapter_dropdown`, and the drag-area playback-panel open consults first and early-returns on.
  Must compose with — not disturb — the existing animation guards (flow anim, theme fade, `_switch`
  state, the sidebar queued-open handoff).

---

## What is NOT the cause (ruled out)

- Not the new shortcuts dispatcher — the collisions reproduce mouse-only (`l` here is just a fast way
  to trigger `_open_library_flow`; a sidebar Library-button click into a speed-button click collides
  identically). Shortcuts make it *easier to hit the timing window*, they don't introduce it.
- Not autorepeat — single discrete presses/clicks, not held keys.
