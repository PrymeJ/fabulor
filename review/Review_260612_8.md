## 3. Action button `delete`/`save`/`save_error`/`check` state machine — (a)* correct, no dead ends (one transient clobber)

`_set_action_mode(mode)` ([tag_manager.py:638-661](src/fabulor/ui/tag_manager.py#L638-L661)) is the single mutator; `self._action_btn_mode` is the state. Four modes (the audit's "trash/save/check" plus `save_error`). Transition map (all `_set_action_mode` sites):

| From → To | Trigger | file:line |
|-----------|---------|-----------|
| (entry) → `delete` | `_open_tag` | [525](src/fabulor/ui/tag_manager.py#L525) |
| `delete` → `save` | type, text ≠ original (`_on_tag_name_changed`) | [628](src/fabulor/ui/tag_manager.py#L628) |
| `save`/any → `delete` | type back to original (`_on_tag_name_changed`) | [630](src/fabulor/ui/tag_manager.py#L630) |
| `save`/`save_error` → `check` | click → `_on_rename` success | [620](src/fabulor/ui/tag_manager.py#L620) |
| `save`/`save_error` → `save_error` | click → `_on_rename` failure (name exists) | [624](src/fabulor/ui/tag_manager.py#L624) |
| `check` → `delete` | `singleShot(2000)` after success | [622](src/fabulor/ui/tag_manager.py#L622) |
| any → `delete` | `_revert_tag_name` (Esc / outside click) | [596](src/fabulor/ui/tag_manager.py#L596) |
| any → `delete` | `_cancel_delete_confirm` | [713](src/fabulor/ui/tag_manager.py#L713) |

**No dead ends.** Every state has a path back to `delete` (the resting state):
- `save` → click (success/fail) or type-back or revert → all reach `delete` or `check`(→`delete`).
- `save_error` → clicking re-runs `_on_rename` ([`_on_action_btn_clicked` handles both `save` and `save_error`](src/fabulor/ui/tag_manager.py#L635)); typing → `save`/`delete` ([628-630](src/fabulor/ui/tag_manager.py#L628-L630)); revert/Esc → `delete`. **Escapable.**
- `check` → auto-returns to `delete` after 2s ([622](src/fabulor/ui/tag_manager.py#L622)). The button is disabled-cursor in `check`/`save_error` ([642-645](src/fabulor/ui/tag_manager.py#L642-L645)) but `_set_action_mode` keeps it `setEnabled(True)` for `check` ([641](src/fabulor/ui/tag_manager.py#L641)) — clicking `check` does nothing (`_on_action_btn_clicked` only acts on `delete`/`save`/`save_error` [632-636](src/fabulor/ui/tag_manager.py#L632-L636)), which is correct.

You **cannot get stuck on "save" with no way back to "trash"**: typing the name back to the original fires `_on_tag_name_changed` → `delete` ([630](src/fabulor/ui/tag_manager.py#L630)), and any outside-click triggers `_revert_tag_name` → `delete` ([590](src/fabulor/ui/tag_manager.py#L590)→[596](src/fabulor/ui/tag_manager.py#L596)). ✅

**`*` transient clobber (minor):** the `check → delete` transition is an **unconditional** `QTimer.singleShot(2000, lambda: self._set_action_mode("delete"))` ([622](src/fabulor/ui/tag_manager.py#L622)) with no guard token. If the user starts typing a *new* rename within those 2s (moving the state to `save` via `_on_tag_name_changed`), the still-pending singleShot fires at T+2s and forces `delete`, overriding the user's in-progress `save`. The button icon would revert to trash while the edit field shows an unsaved change; the next keystroke re-sets `save` ([628](src/fabulor/ui/tag_manager.py#L628)), so it self-heals on input but is a visible flicker. Not a dead end — noted in Findings.

## 4. Delete-confirm cancel paths (timer expiry vs outside click) — (a) correct, identical state

Both cancellation paths converge on the **same** `_cancel_delete_confirm()` method ([tag_manager.py:698-713](src/fabulor/ui/tag_manager.py#L698-L713)):
- **Timer expiry:** `_on_delete_tag` arms `self._cancel_timer.timeout.connect(self._cancel_delete_confirm)` + `start(7000)` ([lines 685-688](src/fabulor/ui/tag_manager.py#L685-L688)).
- **Outside click:** the panel-bg click handler `_on_panel_bg_click` ([598-602](src/fabulor/ui/tag_manager.py#L598-L602)) calls `_cancel_delete_confirm()` when `_confirming_delete`; the locked `_tag_name_edit`/`_detail_dot` `mousePressEvent` overrides set during confirm ([679](src/fabulor/ui/tag_manager.py#L679), [682](src/fabulor/ui/tag_manager.py#L682)) also route to `_cancel_delete_confirm`; `_on_grid_remove` does too ([737-739](src/fabulor/ui/tag_manager.py#L737-L739)).

Because **both paths call the identical method**, cleanup is by construction identical — there is no divergent cleanup. `_cancel_delete_confirm` does the full restore in one place: clears `_confirming_delete`, `_show_reserved("none")`, unlocks the grid, restores both cursors and the `mousePressEvent` handlers on `_detail_dot` and `_tag_name_edit`, clears read-only, **stops and nulls `_cancel_timer`** ([710-712](src/fabulor/ui/tag_manager.py#L710-L712)), and resets the action mode to `delete` ([713](src/fabulor/ui/tag_manager.py#L713)). The timer-expiry path also lands here, so it too nulls its own timer (harmless self-stop). The only difference is *who* invokes it (the timer vs an event), not *what* it does. ✅

## 5. `_set_tag_color` → `_update_list_dot()` only, never `refresh()`; no stale UI — (a) correct

`_set_tag_color` ([tag_manager.py:479-486](src/fabulor/ui/tag_manager.py#L479-L486)):
```python
self.db.set_tag_color(self._current_tag, color_key)
self._show_reserved("none")
self._update_detail_dot(color_key)        # panel-view dot
self._update_list_dot(self._current_tag, color_key)   # list-view dot
self.tag_changed.emit()
```
It does **not** call `refresh()` (the only `refresh()` call in the file is in `_show_list` [548](src/fabulor/ui/tag_manager.py#L548)). It is the only caller of `_set_tag_color` (three dot `mousePressEvent` lambdas: [351](src/fabulor/ui/tag_manager.py#L351), [359](src/fabulor/ui/tag_manager.py#L359)).

**Does `_update_list_dot` + `_update_detail_dot` cover what `refresh()` would?** Yes, for a color change specifically. A color change affects exactly two on-screen elements:
1. **The panel-view detail dot** — handled by `_update_detail_dot` ([507-516](src/fabulor/ui/tag_manager.py#L507-L516)): swaps object name `tag_dot_neutral`↔`tag_dot_colored`, sets/clears the inline color stylesheet, and unpolish/polish to force a restyle.
2. **The matching list-view row dot** — handled by `_update_list_dot` ([488-505](src/fabulor/ui/tag_manager.py#L488-L505)): walks the list rows, finds the row whose `tag_list_name` text matches, swaps the dot's object name + stylesheet, unpolish/polish.

`refresh()` ([400-416](src/fabulor/ui/tag_manager.py#L400-L416)) would tear down and rebuild the entire list from `db.get_all_tags()` — which a color change does not need (the tag set, counts, and names are unchanged; only one dot's color differs). The book grid, counts, and names are untouched by a color change, so skipping `refresh()` leaves nothing stale: `_update_list_dot` targets the one affected row by name match ([496](src/fabulor/ui/tag_manager.py#L496)), and `_update_detail_dot` targets the one panel dot. The `tag_changed.emit()` ([486](src/fabulor/ui/tag_manager.py#L486)) notifies external listeners (library tag-filter chips, etc.) so cross-surface color is propagated without a local rebuild. ✅ Avoiding `refresh()` is correct and complete for this mutation; it also preserves scroll position and avoids a cover-reload flash that a full rebuild would cause.

## 6. `320ms singleShot` reopen — magic number, single location — (a)† correct (single source, documented debt)

**Single site, no duplication:** there is exactly **one** `singleShot(320...)` in the entire codebase — [app.py:1075](src/fabulor/app.py#L1075), in `_on_open_tag_manager_from_detail`:
```python
def _on_open_tag_manager_from_detail(self) -> None:
    self.panel_manager.hide_all_panels()
    QTimer.singleShot(320, self.panel_manager._open_tags_flow)
```
So there is **no "320 in one place, 300 in another" inconsistency** — the value appears once. `PanelManager` itself contains **no** `singleShot(320)`; its sidebar→panel sequencing uses the `sidebar_animation.finished` signal (`_on_sidebar_closed_for_panel` [panels.py:216-228](src/fabulor/ui/panels.py#L216-L228)), a signal-based handoff, not a magic timer. The other 13 `hide_all_panels()` call sites in app.py do **not** reopen a panel with a timer — they only hide.

**The 320 is the documented P6-D debt.** It maps directly to NOTES.md [NOTES.md:549-550](NOTES.md#L549-L550) and [NOTES.md:654-660](NOTES.md#L654-L660): "320ms is a magic number covering the longest panel close animation … the correct fix is an `all_panels_hidden` signal from `PanelManager`." Cross-checking the actual durations confirms the margin reasoning: panel-position close animations are **300ms** (library/stats/settings/sleep/speed/book_detail — [main_window_builders.py:518](src/fabulor/ui/main_window_builders.py#L518),[528](src/fabulor/ui/main_window_builders.py#L528),[549](src/fabulor/ui/main_window_builders.py#L549),[595](src/fabulor/ui/main_window_builders.py#L595); [app.py:476](src/fabulor/app.py#L476),[525](src/fabulor/app.py#L525)), **tags is 200ms** ([main_window_builders.py:537](src/fabulor/ui/main_window_builders.py#L537)); 320 = 300 + 20ms margin. **Severity low**, not medium: a single source with no divergent value means there is **no actual behavioral inconsistency today** — the debt is the fragility (if a panel duration is bumped past 320ms the reopen silently races), not a present-tense bug.

**One latent caveat (informational):** `blur_animation` is **500ms** ([app.py:556](src/fabulor/app.py#L556)) and is part of the close flow + counted by `_any_panel_animating` ([panels.py:544](src/fabulor/ui/panels.py#L544)). The 320ms timer fires while the blur fade is still running. This is benign for the *reopen* (only the panel *position* needs to be off-screen, which completes at 300ms; blur is cosmetic and the new panel's open re-drives blur), but it does mean "all animations finished" is not actually true at T+320ms. The proposed `all_panels_hidden` signal fix should decide whether blur counts toward "hidden" — if it does, the signal would fire ~200ms later than the current timer, a behavior change to be aware of.

## 7. `session_deleted` signal ↔ panel-close timing collision — (c) signal not present

The audit item is premised on "the newly-added `session_deleted` signal (if Prompt 7.5 was completed)." It was **not** added. The Pass 7 finding #9 fix (commit `57a6ef4`) **reused the existing `history_deleted` signal** instead of introducing `session_deleted` — grep for `session_deleted` across `src/fabulor/` returns **nothing**. Per the item's own instruction ("If the signal work hasn't landed yet … skip this item and mark (c)"), this is **(c)**.

**Useful substitute analysis (the live version of the concern):** the actual fix has `_on_history_delete_confirmed` emit `history_deleted` from a `QPropertyAnimation.finished` callback ([book_detail_panel.py:1103](src/fabulor/ui/book_detail_panel.py#L1103)), wired to `stats_panel.refresh_all` + `library_panel.refresh` ([main_window_builders.py:556-557](src/fabulor/ui/main_window_builders.py#L556-L557)). Could that refresh collide with the 320ms reopen timer (item 6)? **No** — they are on disjoint trigger paths: the `history_deleted` refresh fires from the History-tab delete flow *inside* the book-detail panel (no `hide_all_panels`/reopen involved), while the 320ms timer only fires from `_on_open_tag_manager_from_detail` (the "open tag manager from detail" button). The two never run in the same user action. `refresh_all` is a synchronous re-query/repaint of an already-mounted stats panel; it does not start or cancel any panel animation, so even if both were somehow in flight they touch independent state (widget data vs panel geometry animation). No collision. (This is analysis-only; the (c) verdict stands because the specifically-named signal does not exist.)

## Findings requiring follow-up

Notes only — no fixes applied (audit-only task). No `(b)` violations.

1. **[low] `check → delete` auto-revert can clobber an in-progress new rename (item 3).** The `QTimer.singleShot(2000, lambda: self._set_action_mode("delete"))` at [tag_manager.py:622](src/fabulor/ui/tag_manager.py#L622) is unguarded. If the user begins a second rename within 2s of a successful first rename, the pending timer forces the button back to `delete` mid-edit, overriding the `save` state set by `_on_tag_name_changed`. Self-heals on the next keystroke. Candidate fix (not applied): capture a generation token / cancel the pending singleShot when `_on_tag_name_changed` moves to `save`, or store the QTimer and stop it on any new edit. Cosmetic, low priority.

2. **[low — doc] Audit checklist item 2 mis-states the reserved row (item 2).** The reserved row is `setFixedHeight(21)` on a `QWidget` wrapping a `QStackedLayout` ([tag_manager.py:336-339](src/fabulor/ui/tag_manager.py#L336-L339)), not a 32px `QStackedLayout`. No code issue — the height is fixed and un-overridden. Flagging so a future audit's checklist is corrected to "21px wrapper widget."

3. **[informational] 320ms reopen vs 500ms blur (item 6).** The documented P6-D fix (`all_panels_hidden` signal) should explicitly decide whether the 500ms `blur_animation` ([app.py:556](src/fabulor/app.py#L556)) counts toward "all panels hidden." If it does, the signal-based reopen would fire ~180ms later than today's 320ms timer — a deliberate behavior change, not a regression, but worth calling out when the debt is paid down. The panel-*position* close (300ms) is what the reopen actually depends on; blur is cosmetic.
